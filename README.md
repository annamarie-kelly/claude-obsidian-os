# Claude Code + Obsidian: A Personal OS

An AI-native personal operating system built on Obsidian and Claude Code. It's a **capture-and-compound loop** where raw thinking goes in messy and comes out structured, connected, and actionable.

This isn't a note-taking system. It's a thinking system with an AI processing engine on top.

**Built for anyone who thinks for a living** — engineers, investors, analysts, marketers, researchers, operators. If your work involves capturing ideas, tracking commitments, and compounding knowledge across projects and people, this is for you.

## How It Works

```
Capture fast → Triage into domains → Distill into typed notes → Connect → Review → Prune → Compound
```

You dump raw thoughts, meeting notes, ideas, and brain dumps into a single inbox. Claude Code processes them: routing notes to the right **domain** (folder) and **memory type** (episodic / semantic / procedural), extracting patterns as claim-style insights, linking related ideas, surfacing open loops, aging out tactical notes, and running weekly metacognition passes.

Over time, your vault accumulates not just *what you did* but *what you learned*, *how your thinking evolved*, and *how it's indexed so you can find it again*.

## The Two-Axis Model

The vault runs on two axes, not one:

**Axis 1 — Domain** (folder): which area of your life this belongs to.

**Axis 2 — Memory type** (frontmatter `type:`): what *kind* of knowledge this is.

| Memory type | Cognitive role | `type:` values |
|---|---|---|
| **Episodic** | What happened when | `episode` |
| **Semantic** | What I know | `pattern`, `decision`, `failure`, `convention`, `essay`, `reference` |
| **Procedural** | How to do things | `playbook` |

Every note picks one of each. This makes the vault queryable across axes: "show me all failures across building and working," "find all playbooks in `03-Working`," "what episodes produced this pattern." The cost is one frontmatter field per note; the value is compound retrieval.

The highest-value workflow in the vault is **consolidation** — turning episodic captures (what happened on Tuesday) into semantic patterns (what I now know). `/distill` is built around this.

## Vault Structure

```
00-Inbox/          → Single capture point. Everything lands here first.
01-Building/       → Anything being constructed: projects, technical patterns, playbooks
02-Thinking/       → Intellectual life: essays, ideas, references, meta-patterns
03-Working/        → How the business runs: workflow, stakeholders, operations
04-Living/         → Personal life (grows over time)
05-Relating/       → People notes
06-Loops/          → Auto-generated open items dashboard (read-only lens)
07-Archive/        → Done/dead/inactive. Never deleted — just parked.
Templates/         → Note templates
tools/loops-ui/    → Optional Next.js UI for interactive triage + planning
.claude/commands/  → Slash commands for Claude Code
```

Organized by **life domain**, not by content type. Cross-cutting patterns naturally cluster together.

### The `_patterns.md` / `_episodes.md` / `_playbooks.md` Convention

Each folder can have up to three index files — one per memory type:

- **`_patterns.md`** — hand-curated index of semantic notes. `/distill` appends new entries; `/reindex` keeps the links current without flattening your curation.
- **`_episodes.md`** — chronological log of episode notes in that folder (auto-generated by `/reindex`, newest first, grouped by month).
- **`_playbooks.md`** — index of procedural notes in that folder.

These replace traditional Maps of Content (MOCs) with something more dynamic: **living indexes** that grow with your knowledge and split cleanly by memory type.

### Scaling Rule

> Flat until it hurts, then nest. Create subfolders when a domain hits 5+ clustered notes.

## The Commands

Eleven slash commands power the system. Each maps to a different cognitive operation:

| Command | What it does | When to use |
|---|---|---|
| `/setup` | Interactive walkthrough to customize the vault to your role, domains, and tools | Once, on clone |
| `/inbox` | Pull external context from MCP sources (calendar, CRM, task board) into the inbox | ~Once a day |
| `/triage` | Route inbox notes to the right domain + memory type; delegate investigations to background agents | Multiple times/day |
| `/distill` | Classify raw input by memory type (episodic / semantic / procedural) and create typed notes | After learning something |
| `/loops` | Surface all open `- [ ]` items, sorted by inferred priority with domain emojis | As needed |
| `/commitments` | Track what you owe people and what they owe you — grouped by urgency and person | As needed |
| `/find-connections` | Find orphaned notes and suggest missing `[[wikilinks]]` | Maintenance |
| `/conductor` | Scan git worktrees for in-progress branches, stashes, uncommitted work | As needed |
| `/reindex` | Regenerate `_patterns.md` / `_episodes.md` / `_playbooks.md` from frontmatter | After bulk edits |
| `/prune` | Archive notes whose `shelf-life` has expired | Weekly / monthly |
| `/review` | Weekly synthesis: what changed, what's emerging, promote pattern seeds | Once a week |

### `/triage` — The Workhorse

The most-used command. Reads everything in `00-Inbox/` and for each note:

- Determines the **memory type** (episode / pattern / decision / failure / convention / playbook / reference / project update / person note / action item)
- Determines the **domain** (Building, Thinking, Working, Living, Relating)
- Routes the note with proper frontmatter including `type:` and `shelf-life:`
- Extracts action items as `- [ ]` checkboxes (so they surface in `/loops`)
- **Delegates investigation requests to background agents** — captures like "research competitor pricing" or "look into X framework" get kicked off as async research that writes a structured report to `02-Thinking/reports/`
- Cross-links to related notes
- Asks before deleting the original

Key principle: **inbox notes are captured fast with typos — Claude focuses on extracting signal, not correcting spelling.**

### `/distill` — Typed Knowledge Extraction

The metacognition engine. Takes raw input (brain dump, conversation excerpt, meeting notes, session harvest) and produces **typed notes**, not just "patterns":

- A meeting recap becomes one `episode` (what happened) + two `pattern` notes (what you now know) + possibly one `playbook` (how to do this next time).
- A failure retrospective becomes a `failure` note that links back to the `episode` it came from.
- A decision and its rationale become a `decision` note with `**Why:**` + alternatives considered.

Each semantic note still gets a lifecycle: `seed` → `growing` → `evergreen`. The `/review` command prompts you to promote seeds as they mature. The consolidation loop (episode → pattern, confirming existing patterns with new evidence) is the highest-value thing `/distill` does.

Claim-style titles remain the rule for semantic notes:

- "In-process tool calls beat MCP transport for latency-sensitive UIs"
- "Good internal tools hide complexity from users"
- "Knowing is horizontal, understanding is vertical"

### `/loops` — Open Items Dashboard (Priority-Inferred)

Scans every `.md` file for unchecked `- [ ]` tasks and **infers priority from natural language** — no tags or annotations needed:

- **🔴 Now** — due today/tomorrow, "urgent", "blocking", "pressing", overdue deadlines
- **🟡 Soon** — due this week, active projects, no urgency signal
- **🟢 Someday** — "depends on", "brainstorm", "explore", stale items, `@waiting`

Priority is **time-aware**: "due Thursday" on a Wednesday is 🔴 Now, but on a Monday it's 🟡 Soon. Each sub-group gets a **domain emoji** (🔨 Building, 🧠 Thinking, 📋 Working, 🏠 Living, 👤 Relating) so you can see at a glance which area of your life each item belongs to.

Actions: close (auto-closes linked task manager tickets via MCP), move, escalate, delete, bump up/down.

### `/commitments` — Relationship Tracker

A different lens on the same `- [ ]` data, filtered for **interpersonal obligations**. Answers: "What do I owe people? What do people owe me?"

Detects commitment signals (person names, deadlines, "promised", "waiting on", `@waiting`) and groups by urgency:

- **⚠️ Overdue** → **🔴 Due Today/Tomorrow** → **📅 This Week** → **📆 Coming Up** → **🔄 Waiting On Others** → **📋 Undated**

Within each urgency level, items are grouped by person. Use `/loops` for the priority view, `/commitments` for the relationship view.

### `/prune` — Age Out Tactical Notes

Not every observation should live forever. Notes carry a `shelf-life:` field that controls auto-aging:

- **`foundational`** (default) — permanent. Patterns, decisions, conventions, people, projects.
- **`tactical`** — 14-day expiration. Session-specific observations, "for now" notes, working state.
- **`observational`** — 30-day expiration. Time-bounded context, meeting-specific notes, temporary references.

`/prune` scans the vault, builds a report of expired notes, and — after you confirm — moves them to `07-Archive/` preserving folder structure. Nothing gets deleted; expired notes are parked. This keeps `_patterns.md` files from bloating with stale observations while the foundational layer stays legible.

Usage: `/prune --dry-run`, `/prune tactical`, `/prune 00-Inbox/` — scope flags are optional.

### `/reindex` — Regenerate Memory-Type Indexes

After bulk edits, rename-sprees, or `/distill` runs, the per-folder index files can drift from reality. `/reindex` scans every `.md` file's frontmatter, groups by memory type, and regenerates:

- **`_patterns.md`** — preserves your hand-curated sections; only updates links and flags orphans.
- **`_episodes.md`** — fully regenerated, chronological, grouped by month.
- **`_playbooks.md`** — fully regenerated.

The goal is to let you keep the generated indexes trustworthy without losing the human structure in `_patterns.md`.

### `/conductor` — Git Worktree Scanner

For engineers: scans your git repos for in-progress work — feature branches, uncommitted changes, stashes, open PRs. Surfaces forgotten work-in-progress.

### `/find-connections` — Link Discovery

Scans for notes that should be linked but aren't. Finds orphans (zero inbound links) and suggests specific `[[wikilinks]]` to add. This is how the knowledge graph stays connected.

### `/review` — Weekly Metacognition

The synthesis pass. Assumes triage, loops, and (optionally) prune have already been run. Focuses on:

- What domains got attention this week
- What patterns are emerging (suggest claim-style titles)
- What seeds are ready to promote to `growing` or `evergreen`
- What episodes yielded durable patterns
- What notes are orphaned and need connecting
- What felt neglected

### `/setup` — Interactive Onboarding

An interactive walkthrough you run once after cloning. It asks about your role, domains, and tools, customizes `CLAUDE.md`, optionally walks through MCP connector setup (calendar, email, CRM, task board), creates starter content, and runs a live triage demo. Skip any step you don't want.

## Background Agents & Reports

One of the most powerful patterns: **turning quick inbox captures into full research reports without blocking your workflow.**

When `/triage` encounters an investigation request ("investigate X", "look into Y", "research Z"), it:

1. Spins up a background agent to research the topic
2. The agent uses available tools (MCP integrations, web search, codebase search)
3. Writes a structured report to `02-Thinking/reports/` using `Templates/Report.md`
4. Links the report back to the triggering project or note
5. Deletes the inbox note after kickoff

Your inbox can contain both quick tasks ("send invoice") and deep questions ("research how competitors handle onboarding") — triage handles both, routing investigations to async research while you keep working. Reports follow a standard structure: Context, Key Findings, Evidence, Connections, Open Questions. They become permanent reference notes in your vault, cross-linked to the projects and patterns they relate to.

## The Frontmatter Schema

Every note gets:

```yaml
---
created: YYYY-MM-DD
type: pattern | decision | failure | convention | essay | reference
      | playbook
      | episode
      | project | person | loop | seed | trail
status: seed | growing | evergreen | active | someday | done | archived
shelf-life: foundational | tactical | observational
tags: []
---
```

- **`type`** picks a memory type. The three blocks above map to **semantic / procedural / episodic**; the last row is structural (projects, people, loops) and sits outside the memory-type axis.
- **`status`**: semantic notes use the lifecycle `seed → growing → evergreen`. Everything else uses `active | someday | done | archived`.
- **`shelf-life`** controls `/prune`. Default is `foundational` if omitted.

`_patterns.md` files get two extra fields: `scope:` (which domain) and `last-distilled:` (YYYY-MM-DD).

### Record Type Guidance

When a note is about something learned, pick the sharpest type:

- **pattern** — a recurring approach that works (e.g. "X beats Y for Z")
- **failure** — a lesson from something that broke ("X breaks when Y because Z")
- **decision** — why you chose X over Y, with rationale preserved
- **convention** — a rule you follow (formatting, naming, workflow)
- **episode** — what happened on a specific date (meeting, session, incident)
- **playbook** — a step-by-step procedure ("How to X")
- **essay** — longer-form thinking piece
- **reference** — pointer to external material
- **trail** — a named sequence of `[[wikilinks]]` (a reading path with a purpose)

Failures and decisions used to get mashed into "pattern," which conflated *observations about the world* with *hard-won lessons from a specific incident*. The sharper types fix that.

## Note Naming Conventions

- **Patterns**: Claim-style titles — "Good internal tools hide complexity from users"
- **Playbooks**: "How to [verb]" — "How to audit MCP tool reliability"
- **Episodes**: Date-prefixed — "2026-04-17 Planning call with the design team"
- **Failures**: What broke — "Cache invalidation races under concurrent writes"
- **Decisions**: What was decided — "Use cookie auth, not JWT, for the admin console"
- **Projects**: Project name — "Landing Page v2"
- **People**: First name or full name

## The Loops UI (optional visual layer)

`/loops` prints a terminal-friendly snapshot. For longer triage sessions, [`tools/loops-ui/`](./tools/loops-ui/) is a local Next.js app that renders the same `06-Loops/loops.json` as an interactive dashboard:

- **Triage mode** — a queue of new loops you walk through one at a time: accept, park, snooze, or drop. Capacity caps (`P1:<stakeholder>` ≤ 8, `P1:self` ≤ 5) fire before you can add past a ceiling.
- **Plan mode** — drag loops onto a weekday canvas; timeblocks schedule around calendar events pulled from `06-Loops/calendar-today.json`.
- **Focus mode** — one loop at a time, keyboard-driven.
- **Reflect mode** — 30-day pressure heatmap, a weekly stakeholder summary draft, triage-decision stats.

Everything round-trips through Markdown: the UI reads `- [ ]` from your notes and writes `- [x]` back when you close a loop. No database, no auth, no external services — plain file I/O over `06-Loops/`.

Configure the primary stakeholder (the person whose asks get their own P1 bucket and weekly summary) in [`tools/loops-ui/loops.config.json`](./tools/loops-ui/loops.config.json). See [`tools/loops-ui/README.md`](./tools/loops-ui/README.md) for setup.

```bash
cd tools/loops-ui
npm install
node scripts/seed-loops.mjs   # first run only — populates example loops
npm run dev                   # http://localhost:3000
```

## Core Principles

1. **Inbox is sacred.** Single capture point. Don't enforce formatting on raw captures.
2. **Pick a memory type on every note.** One frontmatter field, compound payoff.
3. **Link > tag.** Connections between notes matter more than categories.
4. **Never delete without asking.** Expired notes get parked in `07-Archive/`, not deleted.
5. **Claim-style titles.** Patterns are assertions, not topics. "Authentication is hard" → "JWT cookie auth bridges Django sessions and Next.js cleanly."
6. **Flat until it hurts.** Don't create structure you don't need yet.

## MCP Integration (Optional)

The `/inbox` command is designed to pull from external tools via MCP servers. This is optional — the system works without it. But if you connect your tools, Claude can:

- Pull meeting transcripts and extract action items, decisions, and commitments
- Surface CRM attention flags, stale interactions, deal updates
- Check your task board for assigned tickets
- Cross-reference inbox notes against external data

To customize, edit `.claude/commands/inbox.md` with your MCP tool calls.

## Getting Started

1. Clone this repo into your Obsidian vault directory
2. Install [Claude Code](https://claude.ai/claude-code) if you haven't
3. Open the vault directory in Claude Code and run **`/setup`**

`/setup` will:

- **Ask about you** — role, domains, and tools — and customize `CLAUDE.md` to match
- **Configure MCP connectors** — walk through calendar, email, CRM, task board, docs (or skip)
- **Confirm memory-type conventions** — which types you'll use most, how shelf-life should default
- **Create starter content** — a project, a person, and a seed pattern from your answers
- **Run a live demo** — creates a messy inbox note and triages it so you can see the system work

Everything the setup configures is just markdown files — you can edit any of it later.

### Optional: spin up the Loops UI

If you want interactive triage + week-view planning, follow [the setup in `tools/loops-ui/README.md`](./tools/loops-ui/README.md). It's a local dev server; no deploy.

### Daily / weekly rhythm

**Daily:**
- **Capture** to `00-Inbox/` throughout the day (fast, messy, no formatting needed)
- **`/inbox`** once to pull from connected tools (if configured)
- **`/triage`** to route and process inbox notes (investigations auto-delegate to background agents)
- **`/loops`** to check open items by priority
- **`/commitments`** to check what you owe people

**Weekly:**
- **`/review`** — the metacognition pass (what changed, what's emerging, what's orphaned)
- **`/reindex`** — refresh the per-folder memory-type indexes

**Monthly-ish:**
- **`/prune`** — archive expired tactical/observational notes

**When you learn something:**
- **`/distill`** — classify and type the input, create the right notes

**Maintenance:**
- **`/find-connections`** — keep the knowledge graph connected

## Architecture Decisions

**Why domain × memory type, not domain alone?** Because "where does this belong in my life?" is a natural question, but it doesn't answer "what *kind* of knowledge is this?" One axis gives you a topical index. Two axes give you a queryable memory system — you can ask for all failures in `03-Working`, all playbooks across the vault, all episodes that produced the same pattern. Cost: one frontmatter field. Value: compound.

**Why claim-style titles for patterns?** Because "Authentication patterns" is a topic. "JWT cookie auth bridges Django sessions and Next.js cleanly" is a *reusable insight*. The title IS the takeaway.

**Why `_patterns.md` / `_episodes.md` / `_playbooks.md` instead of MOCs?** MOCs are manually curated indexes that rot. These are distillation targets — `/distill` writes to them, `/reindex` keeps them synced with the underlying notes. Living documents that grow with your knowledge.

**Why separate `/triage` and `/inbox`?** Triage is fast and frequent (process what's already captured). Inbox pull is slow and daily (gather external context). Different cadences, different commands.

**Why `- [ ]` for loops instead of a task manager?** These aren't managed tasks — they're open items that surface during thinking. The real task manager (Linear, Jira, Todoist) handles execution. This handles the thinking layer above it.

**Why infer priority instead of tagging?** Inline tags like `#now` or `#someday` are noise in your notes. The system reads natural language signals ("due Thursday", "blocking", "brainstorm") and the current date to sort automatically. Zero annotation overhead.

**Why two views (`/loops` and `/commitments`)?** Same underlying data, different questions. `/loops` answers "what should I work on?" (priority view). `/commitments` answers "what do I owe people?" (relationship view).

**Why shelf-life instead of auto-archive-by-age?** Because some notes are load-bearing for years (a core decision, a pattern you cite constantly) and some are dead in two weeks (a meeting-specific observation). The note author knows which; the system just honors the declaration. Default to `foundational` — opt into aging, don't opt out.

**Why background agents for research?** Because "look into X" shouldn't block your triage flow. Delegating investigations to async agents means your inbox can contain both 5-second tasks and 5-minute research questions — triage handles both without slowing down.

**Why a separate Loops UI?** `/loops` is perfect for the "what's pressing?" check. The UI is for the longer triage/planning sessions where you want to drag loops onto days, walk through a queue card-by-card, and watch your pressure heatmap over time. Same data, heavier surface.
